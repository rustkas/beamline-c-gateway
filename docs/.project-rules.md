# Project Rules and Conventions

## Documentation Organization

**Rule**: All documentation MUST be organized according to the following structure:

### Project Root (`/`)
- `README.md` - Main project documentation
- `CONTRIBUTING.md` - Contribution guidelines
- `SECURITY.md` - Security policy
- `LICENCE.md` - License
- `CHANGELOG.md` - Version history (optional)

### Documentation Directory (`/docs/`)

**Subdirectories**:
- `docs/readiness/` - Production readiness and decision documents
- `docs/validation/` - Test validation and evidence reports
- `docs/operations/` - Operational guides and procedures
- `docs/architecture/` - Architecture documentation
- `docs/contracts/` - API contracts and specifications

**Index**: `docs/INDEX.md` provides navigation to all documents

**Structure Guide**: `docs/DOCUMENTATION_STRUCTURE.md` defines organization rules

### Working Documents (`/.ai/`)
- Temporary status updates
- Working validation reports
- Internal tracking documents
- **Note**: Not authoritative, frequently updated

### Artifacts (`/artifacts/`)
- Test results and logs
- Timestamped evidence
- Build outputs

## Enforcement

**Before committing**:
1. Check documentation is in correct location
2. Update `docs/INDEX.md` if adding new docs
3. Remove obsolete documents from root

**Monthly maintenance**:
1. Review `.ai/` for obsolete docs
2. Archive or delete superseded reports
3. Ensure `docs/` structure is clean
